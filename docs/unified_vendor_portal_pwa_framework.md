## 2. The reusable framework (how it's "shared, not one-off")

A portal = **Portal Kit (shared) + a thin per-portal glue file + per-portal HTML/manifest/sw/icons**. The Driver portal is the worked example:
- **Shared, zero domain coupling:** `portal-kit.css`, `portal-pwa.js`, `portal-auth.js`, `portal-notify.js`, `portal-shell.js`, `portal-sw-core.js`.
- **Per-portal glue supplies only domain specifics:** the Firestore listeners (`{query, mapDoc}`), the counter/filter definitions, the card `renderCollapsed/renderExpanded` callbacks, the settings accordions, and an inline `STRINGS` (vi/en/es) table.

Deliberately **not** extracted (avoids leaky abstraction): credential UI (phone+PIN vs email+password is too divergent), the card renderer internals (domain-coupled — supplied as callbacks), and i18n strings (each portal ships its own `STRINGS`, mirroring Mobile Barber). This was the conclusion of an adversarial simplicity review; it keeps the kit small and debuggable.

## 3. PWA / manifest / icon changes
- Per-portal **scoped** service worker (`/driver/` like `/mobile-barber/`) — keeps each portal's SW from affecting the rest of the site.
- Manifest omits `name`/`short_name`/`apple-mobile-web-app-title` so each operator names their own Home-Screen icon (the documented iOS pattern).
- Driver icon set is car/shuttle-themed (distinct from the barber icon), navy `#0d2f50` + gold.
- `firebase.json` carve-out makes `/driver/sw.js` non-cacheable so SW updates ship.

## 4. Auth persistence behavior
- `Persistence.LOCAL` forced; the **transient-tolerant gate** signs out only when a *successful* Firestore read proves the account is invalid (`blocked/deactivated/archived/not_found`). On a network/offline/cold-start read **error** it retries with capped backoff and **keeps the session** — fixing the old `driver-admin.html` bug where any read failure bounced the driver to login. Result: logged in until **explicit** logout, surviving refresh / Safari close / Home-Screen relaunch.

## 5. Notification & badge behavior
- **Foreground (primary):** client `onSnapshot` across 5 feeds (rideNotifications offer pool, targeted `bookingOffers`, assigned `bookings`, `travelAssignments`, admin→driver notifications) → in-app popup + chime (after "Enable Alerts" gesture) + **in-app badge** + **home-screen `setAppBadge`** + drawer (mark-read/all) + de-dup; unread count persists across refresh.
- **Background (best-effort):** `sendDriverRidePush` Cloud Function sends a VAPID web push on a new offer, deep-linking to the dashboard (**not** auto-accept — see the 35s caveat below); prunes dead subscriptions.

## 6. iOS Home-Screen notes
- Safari and the installed Home-Screen app have **separate storage** → the driver logs in **once inside the installed app** (expected).
- Home-Screen **icon art is cached at install** — changing icons needs re-add to refresh.
- **Web push + `setAppBadge` require the installed PWA + notification permission** (iOS 16.4+); they're no-ops in a plain Safari tab. Surface an "Add to Home Screen for instant ride alerts" prompt.
- The custom chime only plays while the app is open; backgrounded pushes use the OS sound.
- **35s offer race (known, by design):** ride offers expire in 35s while iOS push can take 5–30s+. Push is therefore **best-effort** and links to the dashboard, not an auto-accept; foreground listeners remain primary. The dispatch window was intentionally **not** changed.

## 7. Security / isolation (Firestore rules, emulator-tested)
Helpers: `isAssignedDriver()` (nested `driver.driverId == uid`), `isPortalVendorUser()` (resource-independent `vendorUsers` exists — **this is what keeps Michael's owner queries working**), `driverUpdateFieldsOk()` (field-pinning allowlist).
- `/bookings` list = `admin | vendorUser | assignedDriver`; update adds field-pinning (a driver cannot forge `paymentStatus`/price/`driver`/`ownerId`).
- `/travel_bookings` list/update = `admin | vendorUser`.
- `/travelAssignments` read scoped to `travel_driver_id == uid` (or admin/owner).
- `/drivers/{driverId}/pushSubscriptions` = self only.
- Barber↔driver isolation already held (different vendor-scoped collection).

## 8. Migration playbook — Food (`vendor-admin.html`), Nails (`salon-admin.html`), Tour, future operators

These are larger, root-hosted monoliths. Migrate incrementally; the Driver portal is the template.

**Per-portal steps:**
1. **Scope folder:** move the portal under its own path (e.g. `/food/`, `/salon/`, `/tour/`) so it gets a clean SW scope; keep the old root URL as a redirect stub (as `driver-admin.html` now does).
2. **PWA:** add `<portal>/manifest.webmanifest` (own scope/theme/icons, omit name/title), `<portal>/sw.js` = config + `importScripts('/portal-kit/portal-sw-core.js')`, and a `firebase.json` rewrite + `/<portal>/sw.js` no-cache header carve-out. Generate a themed icon set (clone `generate_driver_icons.py`).
3. **Auth:** wrap the existing login in `PortalAuth.enableLocalPersistence` + `PortalAuth.guard(...)` with the portal's identity collections (`vendorUsers`/`vendors` + `adminStatus`). Keep the existing credential UI.
4. **Notifications:** replace bespoke notification code with `PortalNotify.init({ listeners, scopeId, ... })` pointing at the portal's order/booking collections; reuse `PortalPWA.subscribePush` + a `sendXxxPush` Cloud Function modeled on `sendDriverRidePush`/`sendMobileBarberBookingPush`; add a `pushSubscriptions` subcollection rule (self-scoped).
5. **Dashboard UX:** render counters/filters/cards via `PortalShell.*` with portal-specific callbacks; move settings into `<details class="pk-accordion">`; load `portal-kit.css` and set the portal's `--pk-accent`.
6. **i18n:** ship an inline `STRINGS` (vi/en/es) table (or reuse `salon-ai-os/i18n.js` where overlap is high) — no hardcoded strings.
7. **Isolation:** add emulator test cases asserting the portal sees only its own data and the owner/admin paths still work, **before** any rules deploy.

**Mobile Barber:** intentionally left untouched (compatible, not rewritten). It already implements these patterns inline; it can adopt the kit later via the same steps if desired — no urgency, zero current risk.

**Michael's unified owner portal:** unchanged and protected; the `isPortalVendorUser()` branch preserves all `ownerId` ride/tour queries + the legacy compat scan (emulator-proven).
